# 仿写学习法 — 设计文档

> 状态：设计稿

---

## 1. 背景

"仿写成熟项目 + 分阶段实现 + 阶段总结"是一套经过实践验证的 AI 辅助学习模式。本方案将其提炼为可复用的方法论，适用于任何技术栈的学习和团队新人 onboarding。

## 2. 目标

| 目标            | 说明                                                   |
| --------------- | ------------------------------------------------------ |
| 个人自学        | 学新语言/新框架时，有一套明确的步骤可循，不迷茫        |
| 团队 onboarding | 新人拿到就能按流程走，产出可预期的学习结果             |
| AI 协作         | 方案内置 AI 交互协议，让 AI 从"问答机"变成"学习引导者" |

## 3. 范围

### 包含

- 完整的五步学习流程
- 每步的交互模板和检查清单
- 可直接复用的文档模板（仿写计划、阶段总结、差距分析）
- AI 交互协议（提问句式、什么场景 AI 该主动/被动）
- 完整的 Go Web 七阶段案例

### 不包含

- AI Skill 的实现（本阶段只出文档，Skill 化作为后续步骤）
- 具体技术栈的教学内容（方案是元方法论，不教 Go/React 等）

## 5. 关键设计决策

### 5.1 为什么是五步不是按项目模块分

方法论按**学习活动**划分（选项目 → 拆阶段 → 实现 → 总结 → 复盘），而不是按项目模块划分（如配置 → 路由 → 数据库）。前者描述的是**学习过程**，后者是**项目内容**。这两套维度是正交的：方法论的五步是"怎么学"，具体应用到某个项目时，你会在"拆阶段"这一步里按项目内容分出子阶段。

### 5.2 为什么把 AI 交互协议单独成章

这是本方案区别于传统学习法的核心差异。大多数学习方案假设"学习者自己搞定"，但我们发现 AI 可以承担：代码解释器、结对编程搭档、Review 者、苏格拉底式提问者等多种角色。明确这些角色和切换时机，是方案可复用的关键。

### 5.3 为什么保留完整案例

纯方法论太抽象。把 Go Web 七阶段作为完整注释案例，读者可以对照看"抽象步骤 → 具体实例"的映射关系，理解后换自己的技术栈套用。
