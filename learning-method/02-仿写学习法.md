# 仿写学习法

> 一套以"仿写成熟项目 + 分阶段实现 + 阶段总结"为核心的 AI 辅助学习方法论。
> 适用于任何技术栈的个人自学与团队 onboarding。

---

## 一、核心理念

### 1.1 为什么仿写比从零学更快

从零学新框架的典型困境：不知道项目结构该怎么搭、不知道配置文件怎么写、不知道异常处理该放哪。这些是"隐性知识"——文档里不会写，只有看过真实项目才能积累。

仿写的价值在于：

- **结构感**：真实项目的目录分层、包依赖关系、初始化链路是经过迭代的，直接复用比从头设计更合理
- **语境**：代码不是孤立的，一个 handler 为什么长这样，要看调用它的 router、它调用的 dbmodel、它返回的 response 格式——仿写迫使你走完整条链路
- **量变到质变**：抄一遍可能只是复制，但**带着问题抄**（为什么这里用指针？为什么这个方法叫这个名？）就会从"知其然"变成"知其所以然"

### 1.2 AI 在仿写中的角色

传统仿写靠自己读源码、查文档、试错，容易卡住。AI 让仿写变成了**交互式学习**：

| 场景               | AI 角色  | 示例                                               |
| ------------------ | -------- | -------------------------------------------------- |
| 看不懂一段代码     | 解释者   | "这个函数签名为什么用 `*int` 而不是 `int`？"       |
| 不知道怎么改       | 引导者   | "我想加一个删除接口，需要改哪几个文件？"           |
| 写完了不确定对不对 | 审阅者   | "帮我 review 这个 handler，有没有遗漏的错误处理？" |
| 想理解设计原因     | 苏格拉底 | "为什么这个项目用全局变量而不是依赖注入？"         |

### 1.3 什么项目适合仿写

**好的仿写对象**：

- 活跃维护、代码质量中等以上的开源项目或个人项目
- 技术栈是你想学的（或者想加深的）
- 有清晰的分层结构（不是一两个大文件撑起来的）
- 规模适中——对一个新手来说，1-3 万行以内比较合适

**不适合仿写的**：

- 过于简单（就是一个文件的事情，仿写没有增量收益）
- 过于复杂（微服务网格这种，仿写成本远高于收益）
- 缺乏文档和注释（你很可能卡在环境搭建上）

## 二、五步流程

### 概览

```
选项目 → 拆阶段 → 实现 → 总结 → 复盘
                  ↑_________|
                    循环
```

每个"实现"步骤内部又是一个微循环：计划一个子任务 → AI 辅助完成 → 验证 → 记录。

### 2.1 选项目

**做什么**：确定要仿写的目标项目，理解它的定位和技术栈。

**产出**：选择理由 + 项目概览笔记（主要模块、依赖关系、启动方式）。

**检查清单**：

- [ ] 目标项目的技术栈是否匹配你想学的东西？
- [ ] 你能把项目跑起来吗？（本地启动、看日志、调一个接口）
- [ ] 项目有合理的分层结构吗？（不是一两个大文件）
- [ ] 你对它的业务领域有基本了解吗？

### 2.2 拆阶段

**做什么**：把仿写过程拆成若干可独立完成的阶段，每个阶段交付一个"能跑"的增量。

**拆分原则**：

- 每个阶段有**明确的目标**（如"阶段四：数据库 + Model，handler 能查 DB 返回数据"）
- 每个阶段结束时有**可验证的产出**（编译通过、curl 返回正确数据、前端页面能展示）
- 阶段之间有**依赖顺序**（先配数据库再写查询，先写查询再做 handler）
- 每个阶段能**独立完成**，不会因为某个阶段卡住就影响后续

**常见的阶段划分方式**（根据项目类型调整）：

```
第一阶段：空壳能跑          → 项目初始化 + 一个 ping 路由
第二阶段：配置 + 日志        → 配置文件 + 日志系统
第三阶段：路由架子            → 目录结构 + 空 handler 占位
第四阶段：数据库 + 数据层     → ORM 初始化 + 查询/写入层
第五阶段：业务 handler        → 逐个实现 API 接口
第六阶段：基础设施补充        → 中间件、校验、鉴权、日志记录
```

**参考**：具体的阶段划分案例参见第五章。

**产出**：一份**仿写计划**，列出每个阶段的目标、步骤、涉及文件和预期产出。

**检查清单**：

- [ ] 阶段的划分是否按依赖关系排列？
- [ ] 每个阶段的目标是否具体可验证？
- [ ] 有没有某个阶段太大，需要进一步拆细？
- [ ] 每个阶段是否都能产出"能跑的东西"？

### 2.3 阶段实现（含 AI 交互协议）

**做什么**：按仿写计划的当前阶段，在 AI 辅助下逐步实现。

**标准实现循环**：

```
1. 理解目标（看原项目的对应代码，问 AI"这段代码在做什么"）
2. 规划改动（本次要改哪些文件、改什么）
3. 执行改动（写代码/修改文件）
4. 验证（编译/运行/curl）
5. 记录（踩了什么坑、有什么疑问、下次注意什么）
```

**AI 交互协议**：

- **提问比请求更有效**：不要说"帮我写一个删除接口"，而是说"原项目的删除接口是怎么实现的？我想在 lab 里加一个类似的，需要改哪些文件？"
- **让 AI 解释而不是代劳**：核心目的是你学会，不是代码完成。当你觉得 AI 直接帮你写了代码时，停下来说"解释一下你刚才写的这段代码"
- **验证闭环**：每次 AI 给你代码后，问"我怎么验证这段代码是对的？"
- **卡住了先分析**：跑不通的时候，把错误信息完整给 AI，而不是盲改

### 2.4 阶段总结

**做什么**：每个阶段完成后，写一篇简短的阶段总结。

**总结模板**：

````markdown
# 阶段X：{阶段名称}

> 日期：YYYY-MM-DD
> 目标：{阶段目标}

## 新增/修改文件

```
{文件变更列表}
```

## 核心思路

{这个阶段做了什么，为什么这么做}

## 关键代码/决策解释

{值得记录的技术细节、踩坑、选择理由}

## 学到的东西

{新概念、新工具、新的理解}

## 验证

```bash
{验证命令}
```

## 下一步

{下一个阶段的目标}

## 概念速查

{本阶段涉及的新概念清单，方便回顾}
````

**检查清单**：

- [ ] 所有新增/修改的文件都列出来了吗？
- [ ] 关键决策和踩坑记录了吗？
- [ ] 下一个阶段的目标明确了吗？

### 2.5 项目复盘

**做什么**：全部阶段完成后，做一次整体回顾和差距分析。

**复盘内容**：

- **回顾各阶段**：每个阶段完成了什么，掌握的能力
- **Lab 项目全景**：完成后的项目结构概览
- **差距分析**：对比真实生产项目，列出
  - 已经能做的事
  - 需要查文档才能做的事
  - 还没接触过的能力
- **下一步建议**：想从零建一个上线项目，还需要补充什么

**差距分析模板**参见第三章。

## 四、AI 交互协议

本协议是仿写学习法的核心——**指导学习者如何有效地与 AI 协作**。

### 4.1 提问句式模板

| 场景     | 低效提问               | 高效提问                                                                               |
| -------- | ---------------------- | -------------------------------------------------------------------------------------- |
| 理解代码 | "这段代码是什么意思？" | "这个函数的参数为什么用 `*int` 而不是 `int`？使用上有什么区别？"                       |
| 规划改动 | "帮我加一个删除功能"   | "我想在原项目已有的增改查基础上加删除，需要改哪几个文件？每个文件要改什么？"           |
| 调试     | "报错了，怎么回事"     | "运行时报了 `{错误信息}`，第 X 行的 `{代码片段}` 执行时报的，可能是什么原因？"         |
| 验证     | "这样写对吗？"         | "写完这个 handler 后，我怎么用 curl 验证它正常工作？成功和失败场景分别应该返回什么？"  |
| 深入理解 | "ORM 是什么？"         | "GORM 的 `First` 和 `Find` 有什么区别？什么场景用哪个？如果查不到记录分别会返回什么？" |

### 4.2 场景决策：AI 主动还是被动

| 场景        | AI 应主动 | AI 应被动 | 说明                                              |
| ----------- | --------- | --------- | ------------------------------------------------- |
| 学习阶段    |           | ✅        | 你决定下一步学什么，AI 按你的节奏来               |
| 调试        | ✅        |           | 当你反复报错找不到原因时，AI 可以主动追问更多信息 |
| 代码 Review | ✅        |           | AI 可以主动指出你代码里的问题                     |
| 规划设计    |           | ✅        | AI 提供选项，你做决策                             |
| 概念解释    |           | ✅        | 你问什么，AI 答什么，不跑题                       |

### 4.3 验证闭环

这是一个关键习惯：AI 给出答案后，你要形成一个**验证回路**。

```

AI 给出答案
↓
你理解答案（不懂就问）
↓
你动手实践/验证
↓
验证结果反馈给 AI（"按你说的改了，还是报错"或"可以了"）
↓
AI 根据反馈调整或确认完成
↓
你记录学到的东西

```

**典型错误**：AI 给了一段代码，你直接复制粘贴跑通了就过了——没有理解、没有验证、没有记录。

### 4.4 AI 使用红线

- **不要让 AI 整个文件重写**——即使产出了你也学不到什么。让 AI 给 diff 或逐段解释
- **不要让 AI 代替你读错误信息**——先把错误自己读一遍，有不懂的再问
- **定期让 AI 考你**——"刚才我们改了三个文件，你说一下每个文件改了什么、为什么改"

### B. 常见问题

**Q: 仿写和抄袭有什么区别？**
A: 仿写是学习手段，不是产出目标。你在仿写过程中理解架构决策、掌握 API 用法、踩坑并记录——这些东西会变成你自己的能力。仿写完成后你不可能交付和老项目一样的代码，因为你的理解会促使你做出不同的设计选择。

**Q: 仿写的项目结构和原项目差异太大怎么办？**
A: 正常。仿写的目标是**学原理**不是**像素级复刻**。如果原项目的某个设计不适合你的场景（比如它的配置来自 Apollo，你本地没有），那就换成你自己的方案（配置来自 yaml）。差异本身也是学习点。

**Q: 每个阶段大概需要多久？**
A: 取决于项目复杂度和你的基础。Go Web 七阶段中，简单阶段（阶段一～三）一天一个，复杂阶段（阶段六～七）两天一个。建议每个阶段不超过三天，太长说明阶段拆得太粗。
