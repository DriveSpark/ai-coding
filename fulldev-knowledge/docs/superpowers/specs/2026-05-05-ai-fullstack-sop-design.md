# 通用 AI 全栈开发 SOP V1 设计稿

## 1. 背景

当前工作场景是已有项目的全栈开发，而不是从 0 到 1 的新系统建设。需求通常从产品文档开始，实际交付会同时涉及 Android、PC Web、后端 Java 服务，以及已有的业务边界、状态流转和上下游依赖。

对开发者而言，最核心的困难不是“不会写某个服务”，而是：

- 拿到需求时，不知道它会落到哪些服务
- 不知道应该先从前端入口还是后端服务入手
- 不知道怎样让 AI 稳定地产出可执行方案
- 不知道如何把一次实践沉淀成下次更快的工作方式

因此，这套 SOP 的目标不是替代技术判断，而是建立一条稳定的工作路径，让开发者能够从一份需求文档出发，快速定位链路、收敛方案、推进交付，并持续沉淀认知。

## 2. 目标

这套 SOP V1 主要解决以下问题：

- 从需求文档出发，在 30 分钟内初判涉及哪些端、哪些服务
- 在半天内产出可执行开发方案
- 在 1 天内摸清主要改动点、风险点和联调依赖
- 尽量把“先问同事”变成“先自己推进到 70%，再带问题确认”

## 3. 适用范围

适用场景：

- 已有项目中的需求开发
- 需求来源主要是产品文档或口头需求
- 开发者同时承担 Android、PC Web 或部分全栈工作
- 可访问前后端代码，但对后端服务架构不够熟悉
- 需求类型为页面、数据展示、业务规则混合型

暂不重点覆盖：

- 从 0 到 1 的全新系统设计
- 大规模架构重构
- 纯基础设施类或纯算法类需求

## 4. 总体思路

这套 SOP 采用“双向夹逼型”工作方式：

- 从前端入口反推接口和后端服务
- 从业务域归属正推主服务和上下游
- 将两边证据合并为需求落点图
- 基于证据让 AI 收敛为开发方案

主链路如下：

`需求文档 -> 需求解码 -> 双向定位 -> 方案收敛 -> 执行准备 -> 交付执行 -> 复盘沉淀 -> 更新 SOP`

## 5. V1 主流程

V1 主流程收敛为 4 个阶段、8 个动作。

### 5.1 阶段一：快速定向

目标：30 分钟内完成大方向判断，不追求一次看全。

动作 1：重写需求

将需求文档翻译成工程语言，至少写清以下内容：

- 谁在使用
- 在哪个端触发
- 做什么动作
- 期望系统返回什么结果
- 成功标准是什么
- 当前未知项是什么

动作 2：找前端承载点

优先定位需求会落在哪些前端位置：

- Android 页面
- PC Web 页面
- 弹窗
- 列表
- 详情页
- 表单
- 流程入口

本阶段产出：

- 需求摘要卡
- 前端承载点列表

### 5.2 阶段二：链路定位

目标：将“感觉会改哪里”变成“有依据地判断会改哪里”。

动作 3：从前端反推接口

围绕前端承载点回答以下问题：

- 这是新增、修改，还是复用现有页面能力
- 变化发生在哪些元素上，例如列表、按钮、字段、校验、状态文案
- 这些变化更像是只改展示、需要新字段、需要新接口，还是需要新规则
- 现有前端代码里是否存在最接近的 API 调用点

本动作的目标是得到：

- 页面承载点
- 可能接口形态
- 变化类型

动作 4：从业务域正推主服务

优先判断这件事本质归谁负责，而不是先平铺所有服务。

常见判断方向：

- UC：用户身份、登录、认证、权限、账户基础资料
- 钱包：充提币、地址、链上动作、币种账户、账本
- 资管：资产划转、资产汇总、可用与冻结、账户资产视图
- 交易：下单、撤单、成交、持仓、委托、交易规则、撮合相关状态
- MIS：后台配置、运营开关、审核流程、管理台操作

判断原则：

`谁对这个业务结果负责，谁就是主服务候选。`

动作 5：补上下游服务

主服务初步确定后，再固定检查以下 5 件事：

- 谁调用主服务
- 主服务依赖谁
- 数据落在哪
- 状态是否联动
- 是否存在异步链路，例如 MQ、定时任务、通知、风控校验

本阶段产出：

- 需求落点图 V1

建议格式：

- 前端承载点
- 变化类型
- 可能接口
- 主服务
- 上下游服务
- 数据影响
- 状态影响
- 异步影响
- 待验证点

### 5.3 阶段三：方案收敛

目标：在半天内形成可执行方案，而不是停留在零散分析里。

动作 6：找相似实现

至少从以下 3 个角度寻找相似实现：

- 相似页面或交互模式
- 相似接口结构
- 相似业务状态流转或校验逻辑

原则：

`优先复用项目中的已知模式，不从 0 开始设计。`

动作 7：让 AI 基于证据出方案

在让 AI 产出方案前，先准备最小证据包：

- 需求摘要卡
- 需求落点图 V1
- 前端承载点
- 相似实现
- 待验证点

AI 使用顺序建议：

1. 先复述需求目标、涉及端、主服务候选、上下游候选、待验证点
2. 再归纳已有实现模式，包括页面承载、接口风格、状态流转、校验逻辑
3. 最后输出建议方案

方案输出建议固定字段：

- Android 改动点
- PC Web 改动点
- 接口建议
- 后端改动点
- 风险点
- 待确认项
- 联调顺序建议

动作 7 后需增加一次人工“去幻想校验”：

- 相似实现是否真实存在
- 主服务判断是否符合业务归属
- 新字段和新接口是否能复用已有结构
- 风险点是否具体有效，而不是泛泛而谈

本阶段产出：

- 开发方案单
- 风险清单
- 待确认问题单

### 5.4 阶段四：执行准备

目标：开始开发前，把顺序、依赖、验收和主要风险定清楚。

动作 8：形成执行与验证清单

建议按以下四步准备：

第一步：拆执行顺序

- 哪些必须先做
- 哪些可以并行
- 哪些必须等联调

建议顺序：

- P0：确认接口和字段口径
- P1：主服务或核心逻辑实现
- P2：Android 和 PC Web 并行改造
- P3：联调与异常分支验证
- P4：回归与验收

第二步：列联调依赖

至少检查：

- 接口是谁提供
- 字段口径谁拍板
- 是否需要 Mock 数据
- 哪些场景必须真实环境验证
- 是否依赖配置、消息、审核流或其他服务

第三步：定验收标准

从以下 4 个角度确认：

- 功能验收
- 数据验收
- 异常验收
- 兼容验收

第四步：做一次开工前风险扫面

固定检查：

- 主服务归属是否可能判断错误
- 字段口径是否存在歧义
- 是否漏掉状态流转
- 是否漏掉异步影响
- 是否漏掉后台或 MIS 配置影响
- 是否误把展示问题当成纯前端问题

本阶段产出：

- 执行顺序
- 联调清单
- 测试策略
- 验收清单
- Code Review 检查点
- 开工前风险检查结果

## 6. 模板库设计

这套 SOP 配套一组轻量模板，重点是让流程能每天复用。

### 6.1 需求摘要卡模板

建议字段：

- 需求名称
- 来源
- 用户对象
- 端别
- 用户动作
- 期望结果
- 成功标准
- 未知项
- 初步风险

### 6.2 需求落点图模板

建议结构：

`前端承载点 -> 可能接口 -> 主服务 -> 上下游服务 -> 数据影响 -> 消息或状态影响 -> 待验证点`

### 6.3 开发方案单模板

建议字段：

- Android 改动点
- PC Web 改动点
- 后端改动点
- 接口定义建议
- 复用的相似实现
- 联调依赖
- 测试策略
- Code Review 状态
- 风险点
- 待确认问题

### 6.4 AI 提示词模板

建议至少维护以下 5 组：

1. 需求解码提示词
2. 服务定位提示词
3. 相似实现搜索提示词
4. 开发方案生成提示词
5. 复盘沉淀提示词

### 6.5 服务认知卡模板

建议字段：

- 服务名
- 核心职责
- 责任人
- 最近验证日期
- 常见入口
- 对外接口类型
- 上下游依赖
- 核心数据对象
- 常见风险
- 相似需求案例

### 6.6 SOP 更新记录模板

建议字段：

- 更新日期
- 更新背景
- 原流程问题
- 新增或修改内容
- 修改原因
- 后续观察点

## 7. 持续迭代机制

这套 SOP 不是一次性文档，而是一套持续增强的工作系统。建议从以下 4 个层面迭代。

### 7.1 按需求迭代

每完成一个需求后，做一次超轻量复盘，只回答以下 4 个问题：

- 这次最有效的一步是什么
- 这次最浪费时间的一步是什么
- 哪个判断一开始错了，后来才纠正
- 下次遇到类似需求，SOP 哪一步要改

### 7.2 按类型迭代

当类似需求累计后，逐步形成子 SOP，例如：

- 列表字段类需求
- 资产或钱包类需求
- 交易规则类需求
- 后台配置类需求
- 登录、权限、用户类需求

### 7.3 按服务迭代

随着接触 UC、钱包、资管、交易、MIS 等服务，不断补充服务认知卡，包括：

- 常见入口
- 常见调用方向
- 核心对象
- 容易误判的地方
- 已有案例

### 7.4 按 SOP 版本迭代

建议以版本方式维护：

- V1：重点解决需求定位问题
- V2：强化交付、联调与风险控制
- V3：强化服务认知积累和团队协作

每次更新都记录：

- 原来哪里不好用
- 这次改了什么
- 后续如何验证它是否更有效

## 8. 推荐落地方式

为了尽快形成正循环，建议按以下方式启动：

1. 先使用这份主流程作为日常工作骨架
2. 补出第一版模板库
3. 在下一次真实需求中强制跑完整个 SOP
4. 需求完成后立即做第一次复盘
5. 根据真实反馈更新到 V1.1

原则：

- 先能用，再优化
- 先沉淀实践有效的动作，再扩充体系
- 先保证主流程稳定，再逐步细化分类 SOP

## 9. 下一步

这份设计稿确认后，下一阶段建议做两件事：

1. 基于本设计稿，产出可直接复制使用的模板库
2. 挑一个真实需求，完整演练一次 SOP，验证 V1 的有效性并产出 V1.1 更新项
